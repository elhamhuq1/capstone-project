# S01: Project Foundation + Ollama AI Proof — Research

**Date:** 2026-03-17

## Summary

S01 is a greenfield scaffold + integration proof. The repository is empty — no code, no `package.json`, no framework setup. The slice must produce a working Next.js 14+ App Router project with TypeScript, Tailwind CSS, SQLite via Drizzle ORM + better-sqlite3, and a streaming `/api/chat` endpoint that forwards to Ollama/Llama 3 8B with an advisory system prompt. This slice supports **R005** (advisory AI chat) and produces the foundation consumed by all downstream slices (S02–S05).

The primary risk is the advisory system prompt quality (R005) — the AI must give guidance and explain issues by default, never volunteering corrected text unless explicitly asked. This is the core experimental control. Secondary risk is that **Ollama is not currently installed** on this machine — it must be installed and the `llama3` model pulled before the proof can run. The Ollama JS client (`ollama` npm package) has a clean async streaming API that maps directly to a Next.js App Router `ReadableStream` response.

The recommended approach is: scaffold first (known pattern), then database setup (Drizzle + better-sqlite3, minimal initial schema), then Ollama integration (helper + system prompt + API route), then prove it end-to-end with a real Llama 3 8B response. The proof is complete when `curl` or a simple test page can POST to `/api/chat` and receive a streaming advisory response from the real model.

## Recommendation

Build in four sequential steps: (1) Next.js scaffold with all dependencies, (2) Drizzle + better-sqlite3 database setup with an initial schema, (3) Ollama integration library + advisory system prompt, (4) streaming API route + proof with real Ollama responses. Keep the proof minimal — the API endpoint and a simple test page that shows streaming text. Do NOT build UI components, participant flows, or study logic — those belong to S02+.

Use `better-sqlite3` as the SQLite driver (not `libsql`) because it's synchronous, battle-tested in Node.js, explicitly mentioned in the boundary map, and needs no remote database features. Use the `ollama` npm package (not raw fetch) for the Ollama client — it handles streaming natively via AsyncGenerator.

Ollama must be installed as a prerequisite: `curl -fsSL https://ollama.com/install.sh | sh` then `ollama pull llama3`. If Ollama is unavailable at build time, the app should still start — the proof step is the only part that requires a running Ollama server.

### Build Order

1. **Scaffold Next.js project** — `npx create-next-app@latest` with TypeScript, Tailwind, App Router, `src/` directory. Then install additional deps: `ollama`, `better-sqlite3`, `@types/better-sqlite3`, `drizzle-orm`, `drizzle-kit`. This unblocks everything.

2. **Database setup** — Create `src/lib/db/index.ts` (Drizzle init), `src/lib/db/schema.ts` (minimal schema), `drizzle.config.ts`. Run `npx drizzle-kit push` to create tables. This proves the DB layer works and is consumed by S02–S05.

3. **Ollama integration** — Create `src/lib/ollama.ts` (chat helper) and `src/lib/prompts.ts` (advisory system prompt). This is the highest-risk work — the system prompt must produce advisory behavior. Build and test in isolation before wiring to the API route.

4. **API route + proof** — Create `src/app/api/chat/route.ts` (streaming POST handler) and `src/app/test-chat/page.tsx` (proof page). Test with real Ollama responses. This is the slice's definition of done: real Llama 3 8B advisory responses streaming to the browser.

### Advisory System Prompt Design (R005)

The system prompt is the most critical artifact. It must enforce:
- **Default behavior**: Identify issues, explain why they're problems, suggest strategies — but do NOT provide corrected text
- **Escalation behavior**: If the user explicitly asks for corrected/rewritten text, provide it
- **Context awareness**: The prompt should reference the specific writing sample being edited

Recommended prompt structure:
```
You are a writing advisor helping a student improve their essay. Your role is to guide, not to do the work for them.

When the student asks for help with their writing:
- Identify specific issues (grammar, clarity, structure, word choice, flow)
- Explain WHY each issue is problematic
- Suggest strategies or approaches for improvement
- Use examples from their text to illustrate points, but do not rewrite their sentences

IMPORTANT: Do NOT provide corrected sentences, rewritten paragraphs, or ready-to-paste text unless the student explicitly asks you to "rewrite," "correct," "fix," or "give me the exact wording." Your default is to teach and guide, not to edit.

The student is working on the following writing sample:
---
{writingSample}
---
```

This needs real-model testing — Llama 3 8B may need additional reinforcement to avoid volunteering corrections.

### Streaming Pattern

The API route converts Ollama's AsyncGenerator to a Web `ReadableStream`:

```typescript
// Pseudocode for route.ts
const stream = await chatWithOllama(messages, systemPrompt);
const readableStream = new ReadableStream({
  async start(controller) {
    const encoder = new TextEncoder();
    for await (const chunk of stream) {
      controller.enqueue(encoder.encode(chunk.message.content));
    }
    controller.close();
  }
});
return new Response(readableStream, {
  headers: { 'Content-Type': 'text/plain; charset=utf-8' }
});
```

### Verification Approach

1. **Scaffold verification**: `npm run dev` starts without errors, `http://localhost:3000` renders the landing page
2. **Database verification**: `npx drizzle-kit push` creates `sqlite.db` with expected tables; a simple insert/select in a test script confirms Drizzle queries work
3. **Ollama verification**: `curl http://localhost:11434/api/tags` confirms Ollama is running and `llama3` model is available
4. **Integration proof**: `curl -X POST http://localhost:3000/api/chat -H 'Content-Type: application/json' -d '{"messages":[{"role":"user","content":"Help me improve this sentence: The dog was very big and it was running fast."}]}'` returns a streaming advisory response (guidance, not a corrected sentence)
5. **Browser proof**: The test-chat page at `/test-chat` shows streaming text appearing word-by-word from Llama 3 8B

## Constraints

- **Node.js 20.20.0** is the runtime — no Bun, no Deno. `better-sqlite3` compiles native addons via node-gyp.
- **`better-sqlite3` is a native module** — Next.js webpack will fail to bundle it unless `serverExternalPackages: ['better-sqlite3']` is set in `next.config.ts`. This is a hard requirement.
- **Ollama must be running on localhost:11434** — the `ollama` npm package defaults to this. The app should gracefully handle Ollama being unavailable (return a clear error, don't crash).
- **Llama 3 8B must be pre-pulled** (`ollama pull llama3`) — the model is ~4.7GB. The proof step fails without it.
- **Ollama is not currently installed** on this machine. Installation requires: `curl -fsSL https://ollama.com/install.sh | sh` then `ollama serve` (or it auto-starts via systemd) then `ollama pull llama3`.

## Common Pitfalls

- **Webpack bundling `better-sqlite3`** — Next.js tries to bundle all imports for server components. Native modules like `better-sqlite3` break webpack. Fix: add `serverExternalPackages: ['better-sqlite3']` to `next.config.ts`. Without this, `npm run dev` will crash with a webpack resolution error.
- **Ollama AsyncGenerator not closed on client disconnect** — If the browser closes mid-stream, the API route's `for await` loop keeps running. Use `ReadableStream`'s `cancel()` callback to abort the Ollama stream. Otherwise, abandoned requests consume model GPU time.
- **System prompt too weak** — Llama 3 8B may ignore advisory constraints and volunteer corrections anyway. Test with multiple prompt variations. Add reinforcement like "Remember: guide, don't rewrite" at key points. The proof step must validate this behavior.
- **SQLite file path** — `better-sqlite3` uses a relative path by default. In Next.js, the CWD is the project root, so `sqlite.db` ends up at the project root. This is fine for localhost, but make sure `.gitignore` includes `*.db`.

## Open Risks

- **Ollama installation may require sudo** — the install script needs root access for systemd service setup. If the build environment restricts sudo, Ollama must be installed manually or pre-installed.
- **Llama 3 8B latency on CPU** — if this machine has no GPU, responses may be very slow (30+ seconds for a paragraph). This could affect the study flow in later slices but doesn't block the proof — even a slow response proves the integration works.
- **Advisory system prompt may need iteration** — the first version likely won't be perfect. The proof step should include 3-5 diverse test prompts to validate advisory behavior. If Llama 3 8B consistently ignores the advisory constraint, consider stronger prompt engineering or a different model.
